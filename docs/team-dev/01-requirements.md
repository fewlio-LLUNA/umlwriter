## 1. 目的・背景
- 描いたクラス図を、GitHub / Notion / Obsidian などがそのまま描画できる **Mermaid 記法の Markdown** として持ち出したい。
- 既存の書き出しは JSON（このアプリでしか読めない）と SVG / PNG（画像で、テキスト差分が取れない）の2つ。ドキュメントに埋めて **Git で差分が追える** 形が無いのを埋める。
- 画像を貼るのではなく Mermaid ソースで持ち出せると、貼った先で再レイアウト・微修正ができる。

## 2. ユーザーと利用シーン
- 想定ユーザー: クラス図を描く本人（Miliba）。
- 主な利用シーン: 設計メモや README に図を貼る。Mermaid ブロックを貼り付けて、そのまま設計ドキュメントの一部にする。

## 3. スコープ
### やること（In）
- ツールバーの「Markdown 書き出し」ボタン1つで `class-diagram-YYYYMMDD.md` を即ダウンロードする。
- 出力は **見出し1行＋```mermaid フェンス** だけのシンプルな .md（本文の解説は付けない）。
- クラス・属性・操作・ステレオタイプ・可視性・static / abstract を Mermaid の対応記法へ写す。
- 関連 6 種を Mermaid の対応する矢印へ写す。多重度・関連ラベルも出す。
- パッケージは **座標の包含判定** で所属クラスを決め、Mermaid の `namespace` として出力する。
- Mermaid のクラス名規則に反する名前（スペース・括弧など）を **サニタイズ**し、表示名は別途保つ。

### やらないこと（Out）
- Mermaid → 図の **読み込み（インポート）**。今回は書き出しの一方向のみ。
- 書き出しプレビュー画面・クリップボードへのコピー導線（今回はダウンロードのみ）。
- Mermaid で素直に表現できない情報の保全（後述の F8 の通り **捨てる**）。往復できる保存は既存の JSON 書き出しが担う。
- ノート（`NoteNode`）の出力。現状ノートを作る UI が無く、常に空配列のため。
- Mermaid 以外の記法（PlantUML など）への書き出し。
- 出力する Mermaid のテーマ・レイアウト指定（`direction` など）の設定 UI。

## 4. 機能要件
| ID | 機能 | 概要 | 優先度 |
|----|------|------|--------|
| F1 | 書き出しボタン | ツールバーから `class-diagram-YYYYMMDD.md` をダウンロード。既存 JSON / 画像と同じ流儀 | MUST |
| F2 | クラス変換 | `class 名前 { … }` に属性・操作を並べる | MUST |
| F3 | 可視性・分類子 | `+ - # ~` をそのまま写し、static は `$`、abstract な操作は `*` を付ける | MUST |
| F4 | ステレオタイプ | abstract / interface / enumeration を `<<…>>` として出す。`none` は出さない | MUST |
| F5 | 関連 6 種 | 下の対応表どおりに矢印へ写す。**向き（source / target）の入れ替えに注意** | MUST |
| F6 | 多重度・ラベル | `Library "1" o-- "*" Book : 蔵書` の形で出す。空なら省略 | MUST |
| F7 | 名前のサニタイズ | Mermaid のクラス名規則に反する文字を `_` に置換し、衝突は連番で回避。元の名前は表示ラベルとして保つ | MUST |
| F8 | 表現できない情報の切り捨て | ロール名・ノードの座標・各種 description は出力しない | MUST |
| F9 | パッケージ → namespace | 枠の矩形に収まるクラスをその namespace に入れる。どの枠にも入らないクラスは namespace の外に出す | SHOULD |
| F10 | 空図の扱い | クラスが 0 件でも壊れた .md を出さない（`classDiagram` だけの有効な図） | SHOULD |

### 4-1. 関連の対応表（向きに注意）
本アプリは `source → target` で意味を固定している（[relationships.md](../../uml-class-diagram-tool/uml-notation/relationships.md)）。
Mermaid は **記号が左側のクラスに付く** ため、汎化・実現は左右を入れ替える。

| kind | 本アプリの意味 | Mermaid 出力 |
|------|----------------|--------------|
| `association` | 単純な結びつき | `source --> target` |
| `aggregation` | source が全体（白ダイヤ） | `source o-- target` |
| `composition` | source が全体（塗りダイヤ） | `source *-- target` |
| `generalization` | target が親 | `target <|-- source` ← **入れ替え** |
| `realization` | target が interface | `target <|.. source` ← **入れ替え** |
| `dependency` | source が target を使う | `source ..> target` |

多重度は矢印の両脇に置く: `source "sourceMultiplicity" --> "targetMultiplicity" target : label`。
汎化・実現で左右を入れ替えるときは、**多重度も一緒に入れ替える**こと。

## 5. 非機能要件
- ブラウザ完結・バックエンドなしの既存構成を維持（Mermaid のレンダリングもしないので、新規依存は追加しない見込み）。
- 変換は `Diagram → string` の **純粋関数** として切り出し、DOM に依存させない（単体テストしやすくするため）。
- 既存の JSON / 画像書き出し・共同編集に回帰を出さない。
- 出力した Mermaid が **実際に描画できる**こと（構文エラーで落ちない）。

## 6. 制約・前提
- 公式ドキュメント（<https://mermaid.js.org/syntax/classDiagram.html>）で確認済みの事実:
  - 可視性は `+` `-` `#` `~`。static は `$`、abstract は `*` を末尾に付ける。
  - ステレオタイプは `<<Interface>>` のように `<<` `>>` で囲む。
  - 多重度は矢印の前後に `"1"` のようにクォートで置く。ラベルは `: LabelText`。
  - **クラス名に使えるのは英数字（Unicode 可）・アンダースコア・ダッシュのみ** → 日本語のクラス名はそのまま使えるが、スペースや括弧は使えない。ここが F7 が要る理由。
  - 操作の戻り型は `)` の後ろに **半角スペース区切り**で書く（コロンではない）。例: `+getBalance() BigDecimal`。
  - 属性の型は **型が先・名前が後**（例: `+String owner`）がドキュメントの書き方。本アプリの表示（`- title: String`）とは逆順になる。
  - ジェネリクスは `List~int~` のようにチルダで囲む（`<` `>` ではない）。
  - `namespace` が使える。
- 既存資産: `src/lib/umlFormat.ts` は**画面表示用**の整形なので、Mermaid 用にそのまま流用しない（型の順序も戻り型の区切りも違うため）。新規に変換モジュールを起こす。
- 新規ファイルは `src/lib/exportMermaid.ts` を想定。ダウンロード処理は `src/lib/jsonIo.ts` の `todayStamp()` と同じ流儀に揃える。

## 7. 受け入れ条件（完成の定義）
- ツールバーのボタンで `class-diagram-YYYYMMDD.md` がダウンロードされる。
- 出力した .md を GitHub か Mermaid Live Editor に貼ると、**構文エラーなく描画される**。
- 描画された図が、アプリ上の図と次の点で一致する: クラス名・属性・操作・可視性・ステレオタイプ・**関連の種類と向き**・多重度。
- 汎化を1本引いた図を書き出すと、Mermaid 上でも「親が上・子が下」の継承として正しく出る（向きが逆になっていない）。
- パッケージ枠に囲んだクラスが namespace の中に、囲んでいないクラスが外に出る。
- クラス 0 件の状態で書き出しても、壊れた .md にならない。
- 既存の JSON / SVG / PNG 書き出しに回帰がない。

## 8. 未確定・要確認事項（承認前に確認）
以下は既定で進める想定。異論があれば指摘してほしい。

1. **属性の型の書き順**（要検証）: ドキュメント準拠なら `+String title` だが、本アプリの表示は `- title: String`。Mermaid はメンバ行を概ねそのまま表示するため `-title: String` も通る**可能性が高いが未確認**。実装時に Mermaid Live Editor で両方試し、**アプリの見た目に近い `-title: String` が通るならそちらを採用**する（通らなければドキュメント準拠にフォールバック）。
2. **表示ラベルの保持**（要検証）: `class Book["注文 (Order)"]` のラベル構文はドキュメントに記載があるが、**対応バージョンが未確認**。使えない場合は、サニタイズ後の名前をそのまま出す（元の名前は失われる）。
3. **static / abstract と戻り型の並び**（要検証）: `+find() Book$` なのか `+find()$ Book` なのか、ドキュメントから確定できない。実装時に実際に描画して決める。
4. **パッケージの包含判定**: クラスの**左上座標**が枠の矩形に入っていれば所属とみなす（クラスの実寸を測らない簡易判定）。複数の枠に重なる場合は**面積が小さい枠＝内側**を優先。
5. **ジェネリクス**: 型に `<` `>` が含まれていたら `~` に置換して Mermaid のジェネリクス記法に寄せる。ただしカンマを含むジェネリクス（`Map<K, V>`）は Mermaid が非対応のため、その場合は置換せず素通しする。
6. **見出し**: .md の1行目は `# クラス図` 固定（図に名前を付ける機能が無いため）。
7. Mermaid → 図の読み込み、プレビュー画面、ノート出力は将来検討（本要件では Out）。
